5. Unconditional DPmix with Stick-Breaking Backend

Unconditional DPmix: Stick-Breaking (SB) Backend

Goal: Estimate density of univariate outcome \(y\) using nonparametric Dirichlet Process mixture with Stick-Breaking backend.

Model: \(y_i | G \sim \sum_{j=1}^{J} w_j K(y_i; \theta_j)\) where weights \(w_j\) follow stick-breaking process

Backend: SB with fixed \(J\) components

Data Setup (Same as CRP)

Code 
# Generate from mixture of gamma distributions
set.seed(42)
component1 <- rgamma(60, shape = 2, rate = 1.5)
component2 <- rgamma(40, shape = 1, rate = 0.5)
y_mixed <- c(component1, component2)

# Visualization
df_data <- data.frame(y = y_mixed)
p_raw <- ggplot(df_data, aes(x = y)) +
  geom_histogram(aes(y = after_stat(density)), bins = 30, alpha = 0.6, fill = "darkorange") +
  geom_density(color = "darkred", linewidth = 1) +
  labs(title = "Raw Data: Mixed Gamma Distribution", x = "y", y = "Density") +
  theme_minimal()

print(p_raw)

Model Specification (SB Backend)

The Stick-Breaking (SB) backend uses a fixed number of mixture components. Unlike CRP which adaptively determines the number of components, SB requires you to specify the exact number upfront.

Key Differences:

  • CRP (Vignette 4):
    • Uses components as truncation level (Kmax)
    • Actual # of components inferred from data (≤ Kmax)
    • Flexible, data-adaptive
  • SB (This Vignette):
    • Uses components as FIXED number
    • Always uses exact number specified
    • Simpler, faster (no label-switching overhead)

Bundle & MCMC Settings

Code 
bundle_sb <- build_nimble_bundle(
  y = y_mixed,
  kernel = "lognormal",
  backend = "sb",
  components = 5,
  mcmc = list(
    niter = 2000,           # Total iterations
    nburnin = 500,          # Burn-in
    nchains = 2,            # Parallel chains
    thin = 1                # No thinning
  )
)

Computational Efficiency

CRP - Adaptive # of components (truncation) - Handles label-switching via NIMBLE - Slower per iteration - Better for unknown complexity

SB - Fixed components - No label-switching overhead - Faster per iteration - Better when component count is known

Key Takeaways

  • SB Backend: Fixed components, simpler inference, faster computation
  • J Parameter: Choose based on expected mixture complexity
  • Trade-offs: SB trades flexibility (CRP) for computational efficiency
  • Model Comparison: Use summary() to compare ESS, Rhat, and convergence
  • Next: Add GPD tail augmentation in vignette 6

When to Use SB vs CRP

Use CRP (Vignette 4) if: - Unknown mixture complexity - Want data-adaptive # components - Willing to pay computational cost - Small to medium sample size

Use SB (this vignette) if: - Known or assumed # components - Need faster MCMC per iteration - Large dataset (SB scales better) - Prefer simpler, more stable inference
