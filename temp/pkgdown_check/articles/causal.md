# DPmixGPD: Causal treatment effects (ATE/QTE) via two-arm outcome models

``` r

library(DPmixGPD)
set.seed(1)
```

DPmixGPD includes a causal workflow for estimating treatment effects on
the *entire* outcome distribution. Rather than restricting attention to
a mean difference, the causal module targets:

- mean-scale effects (ATE),
- quantile-scale effects (QTE), and
- truncated/restricted mean effects (available through
  `ate(..., type="rmean")`).

The causal workflow reuses the same outcome modeling machinery as the
unconditional and conditional vignettes: a Dirichlet process mixture
(bulk) optionally spliced with a generalized Pareto tail (GPD). The
mathematical model for the DPM prior, the GPD density/CDF, and the
bulkâ€“tail splice is developed once in the vignette **â€œDPmixGPD:
Model specification and posterior computationâ€?**. This vignette
focuses on causal identification, how DPmixGPD builds a two-arm model,
and how to compute ATE/QTE from posterior draws.

## Causal setup and identification

### Potential outcomes and estimands

Let $`T\in\{0,1\}`$ denote a binary treatment indicator and
$`Y(0), Y(1)`$ the potential outcomes. The observed outcome is
``` math
Y = T\,Y(1) + (1-T)\,Y(0).
```
For a covariate vector $`\boldsymbol{x}`$, define the conditional CDFs
and quantiles
``` math
F_a(y\mid \boldsymbol{x}) = \Pr\{Y(a)\le y \mid \boldsymbol{X}=\boldsymbol{x}\}, \qquad
Q_a(\tau\mid \boldsymbol{x}) = F_a^{-1}(\tau\mid \boldsymbol{x}), \quad \tau\in(0,1),
```
and the conditional quantile treatment effect
``` math
\mathrm{QTE}(\tau\mid \boldsymbol{x})
=
Q_1(\tau\mid \boldsymbol{x}) - Q_0(\tau\mid \boldsymbol{x}).
```
The conditional mean effect (ATE at $`\boldsymbol{x}`$) is
``` math
\mathrm{ATE}(\boldsymbol{x})
=
\mathbb{E}\{Y(1)\mid \boldsymbol{x}\} - \mathbb{E}\{Y(0)\mid \boldsymbol{x}\}.
```

DPmixGPDâ€™s
[`qte()`](https://arnabaich96.github.io/DPmixGPD/pkgdown/reference/qte.md)
and
[`ate()`](https://arnabaich96.github.io/DPmixGPD/pkgdown/reference/ate.md)
functions compute these treated-minus-control functionals from a fitted
causal object. If you supply `newdata`, you obtain effects evaluated at
those covariate profiles; otherwise DPmixGPD defaults to the training
design $`X`$ when present.

### Assumptions

DPmixGPD follows the standard â€œselection on observablesâ€?
identification strategy (Rubin causal model) ([Rubin
1974](#ref-rubin1974)). The identifying assumptions are:

1.  **Consistency / SUTVA**: the observed outcome equals the potential
    outcome under the received treatment; no interference between units.
2.  **Unconfoundedness**:
    ``` math
    \{Y(0),Y(1)\} \perp T \mid \boldsymbol{X}.
    ```
3.  **Overlap** (positivity):
    ``` math
    0 < e(\boldsymbol{x}) < 1 \quad \text{for covariate values of interest,}
    ```
    where
    $`e(\boldsymbol{x})=\Pr(T=1\mid \boldsymbol{X}=\boldsymbol{x})`$ is
    the propensity score.

Under these assumptions,
``` math
F_a(y\mid \boldsymbol{x}) = \Pr(Y\le y \mid T=a,\boldsymbol{X}=\boldsymbol{x}),
```
so causal effects can be estimated by modeling the *observed*
conditional outcome distribution within each treatment arm.

## DPmixGPD causal strategy: two-arm outcome models

DPmixGPD builds:

- a control-arm outcome model for $`Y\mid T=0`$,
- a treated-arm outcome model for $`Y\mid T=1`$,
- and (optionally) a propensity score (PS) model for
  $`T\mid \boldsymbol{X}`$.

Each outcome arm is a full DPmixGPD model (bulk DPM with optional GPD
tail). You can choose different kernels/backends/tail settings per arm
by passing length-2 vectors for `kernel`, `backend`, `GPD`, and
`components`.

### Why include a propensity score module?

Rosenbaum and Rubin showed the propensity score is a balancing score:
adjusting for $`e(\boldsymbol{x})`$ is sufficient to remove bias due to
observed covariates ([Rosenbaum and Rubin 1983](#ref-rosenbaum1983)).
DPmixGPD offers a lightweight PS model and can pass a PS-derived
covariate into both outcome arms as an additional predictor. This is
optional:

- set `PS = FALSE` to skip propensity score estimation and use only
  $`X`$ in the outcome models;
- set `PS = "logit"` / `"probit"` / `"naive"` to estimate PS and include
  it in the outcome arms.

Important details:

- `ps_scale = "prob"` uses the posterior summary
  $`\hat e(\boldsymbol{x})`$ directly.
- `ps_scale = "logit"` uses $`\mathrm{logit}(\hat e(\boldsymbol{x}))`$
  (often numerically friendlier as a covariate).
- `ps_summary = "mean"` or `"median"` chooses how the posterior for
  $`e(\boldsymbol{x})`$ is summarized before being passed into the
  outcome arms.
- `ps_clamp` clamps $`\hat e(\boldsymbol{x})`$ to
  $`(\varepsilon, 1-\varepsilon)`$ to avoid infinities under the logit
  transform.

## Building and fitting a causal model in code

We illustrate a typical workflow on a small synthetic example. For real
analyses, increase MCMC iterations and check diagnostics as in any
Bayesian mixture model.

### Simulate data with confounding (for demonstration)

``` r

set.seed(2)
N <- 250
X <- cbind(x1 = rnorm(N), x2 = rbinom(N, 1, 0.5))

# Treatment assignment via a propensity score
ps_true <- plogis(-0.2 + 0.9 * X[, 1] - 0.6 * X[, 2])
T <- rbinom(N, 1, ps_true)

# Heavy-tailed positive outcome: treatment shifts the bulk and changes tail risk
base <- exp(0.2 + 0.4 * X[, 1] + 0.2 * X[, 2] + rt(N, df = 3) / 3)
y <- base * exp(0.25 * T)
```

The causal bundle constructor is
[`build_causal_bundle()`](https://arnabaich96.github.io/DPmixGPD/pkgdown/reference/build_causal_bundle.md).
It creates:

- a propensity score bundle (if `PS` is not `FALSE` and $`X`$ is
  present),
- a control outcome bundle,
- a treated outcome bundle.

#### Arm-specific configuration (optional but supported)

You can use different kernels/backends/tail settings across arms:

``` r

cb2 <- build_causal_bundle(
  y = y, X = X, T = T,
  kernel = c("lognormal", "gamma"),      # treated, control
  backend = c("sb", "sb"),
  GPD = c(TRUE, TRUE),
  components = c(6, 6),
  PS = "probit"
)
```

#### Arm-specific priors and link specifications via `param_specs`

`param_specs` works the same way as in
[`build_nimble_bundle()`](https://arnabaich96.github.io/DPmixGPD/pkgdown/reference/build_nimble_bundle.md),
but you may pass either:

- one list (used for both arms), or
- a list with `con` and `trt` entries to override each arm separately.

For example, to let the bulk location depend on $`X`$ in both arms
(conditional outcome distributions), you can use link mode (see the
conditional vignette for full details):

``` r

param_specs <- list(
  bulk = list(
    meanlog = list(
      mode = "link",
      link = "identity",
      beta_prior = list(dist = "normal", args = list(mean = 0, sd = 2))
    ),
    sdlog = list(
      mode = "dist",
      dist = "invgamma",
      args = list(shape = 2, scale = 1)
    )
  ),
  gpd = list(
    tail_scale = list(
      mode = "dist",
      dist = "gamma",
      args = list(shape = 2, rate = 1)
    ),
    tail_shape = list(
      mode = "dist",
      dist = "normal",
      args = list(mean = 0, sd = 0.25)
    )

cb3 <- build_causal_bundle(
  y = y, X = X, T = T,
  backend = "sb",
  kernel = "lognormal",
  GPD = TRUE,
```

## Estimating QTE and ATE

### Quantile treatment effects (QTE)

DPmixGPD computes QTE by:

1.  predicting $`Q_1(\tau\mid \boldsymbol{x})`$ from the treated outcome
    model,
2.  predicting $`Q_0(\tau\mid \boldsymbol{x})`$ from the control outcome
    model,
3.  subtracting draw-by-draw across posterior samples to obtain a
    posterior for the difference.

This is aligned with classical quantile treatment effect definitions
([Firpo 2007](#ref-firpo2007)) and the broader quantile regression
literature ([Koenker and Bassett 1978](#ref-koenker1978)).

## Diagnostics and practical guidance

1.  **Check overlap**. If treated and control units have little overlap
    in propensity score, causal estimands at covariate values outside
    overlap become extrapolation problems regardless of outcome model
    flexibility. Rosenbaumâ€“Rubin diagnostics (PS histograms by group)
    remain useful here ([Rosenbaum and Rubin 1983](#ref-rosenbaum1983)).

2.  **Tail identifiability matters**. If you enable `GPD=TRUE` but only
    a small number of exceedances are effectively informed by the
    splice, the tail parameters can be weakly identified and extreme
    QTEs (e.g., $`\tau=0.99`$) may have wide intervals.

3.  **Start simpler**. If this is your first causal DPmixGPD run,
    consider:

    - bulk-only (`GPD=FALSE`),
    - linking only one bulk location parameter to $`X`$,
    - then incrementally adding tail modeling and more link-mode
      parameters.

4.  **Backend choice**. For causal models that use link-mode parameters
    (conditional outcome distributions), `backend="sb"` is the
    recommended starting point.
